QA Automaton and Mobile:
Build Apps That Work, Wow and Win!

Mobile is moving too fast for your usual develop, test and launch strategy. Make no mistake - the planning and strategy is important but it should be executed in six weeks not six months. The need for speed, growing customer intolerance for underperforming apps and a strategic necessity to scale and adapt to the changing mobile usage scenarios have put test automation center stage in Mobile app planning.

Whether you are in the thick of the mobile race - or getting ready for it, making your apps customer and future ready quickly and accurately can set you up for early success. Here's some information that can help you build a robust and scalable program.

The Dos and Don'ts When Automating Mobile Testing

When done right, QA automation is simple, effective and designed to keep you worry-free. But quite often taking some common missteps can leave you with testing program that is frustrating mess of mismanaged expectations, missed deadlines, incompatible platforms, and questionable ROI. Â Here are some tips on what you should and should not do.

The Dos

Test, Test and Test for Memory-Related Issues/App Performance Read moreRead less

The most significant performance issues with Mobile applications arise from how they interact with the device memory or vice-versa. A minor mismatch and the application will crash - often multiples times in a day, frustrating the users. For e.g., smartphone users cannot close different applications easily. When they move from chatting on whatsapp to checking a travel app to reading an e-book - they are unintentionally running multiple instances concurrently. This will impact your app performance unless you preemptively tested it for such scenarios. Memory defects testing and app optimization for performance in a variety of memory-usage scenarios is nearly impossible to achieve manually.

Implement Continuous Integration Read moreRead less

Your test framework should support a Continuous Integration tool - open source or paid. It saves you time and money over the lifespan of the project as you detect and fix problem early and quickly. No more last-minute chaos as you work in "real-time".

Find out How Much of Your Code is Actually Covered Read moreRead less

In the past-faced development environment of mobile applications, the test coverage you want to achieve is rarely achieved just by adding volumes of test cases. You may write 200 test cases but might still be missing the real case as time-pressures prevent you from running all. Get an analysis of how much of the code is actually covered and plan ahead.

Keep Functional Tests IndependentRead moreRead less

Accurate functional testing is critical for ensuring your app's functions - ability to login, set preferences, completing transactions, etc.,- work accurately, reliably and securely for the end user. Keep each test independent to avoid getting repeated failures. Every time you test a function, you change the entry condition. The next successive test may fail as it may not have the same entry condition.

Ensure Test Framework Supports All Available SDKS in the Market, Especially with Android. Read moreRead less

While important in all cases, SDKs are particularly important when dealing with Android where application versions range widely - from 4.4. to 2.2 - and people use them across different devices. Your test framework must be capable of working on any of the SDKs. You simply cannot get full multi-device of coverage manually.

Think About Rotation. Read moreRead less

With handheld devices, user interaction with your application changes every time they move their device- changing the screen rotation from portrait to landscape to vice versa. (+ expand to show the rest) Your test framework must factor in the variety of ways a user can interacts with your application on a mobile device. Apps that are not optimized rotations face several performance issues.

The Don'ts

Don't Rely on Record and Playback. Read moreRead less

The Record and Playback framework has serious reliability and accuracy problems as it does not perform across platforms. For e.g. in an application rolled out on android, iPhone and iPad - with Record and Playback every time the screen size changes, your parameters change. You'll end up maintaining multiple suites for multiple devices and even then not get proper coverage. Tools like Appium as well are inadequate as your scripts will fail frequently if there is a change in the element parameters like size. It's also a lot more difficult to debug these failures.

Don't Chase Numbers and Loose Focus on Code Coverage. Read moreRead less

Often QA leaders are focused on increasing the volume of test cases but overlook how much code coverage these thousands of test cases are actually giving. The focus should be on getting more code coverage with less tests.

You Cannot and Should Not Automate Everything. Read moreRead less

When creating an automation strategy, your manual testers and automation engineers should work in tandem. You only want to automate scenarios which are passing at least once. Why waste resources on creating automation code for something that is already failing?

Don't Keep Automation in a Silo. Read moreRead less

Do not segregate automation from development and QA. It's inefficient and slows down the rollout. If working in tandem, any changes made by the QA team and developers are immediately visible to automation engineers who can adjust the test code accordingly.

Don't Underutilize The Skills of Your Automation Team. Read moreRead less

Usually automation engineers just report on failures but are not allowed to or discouraged from logging defects. Give them access to defects tracking system and allow them to suggest improvements.

Don't have an Anti-Automation Branching Strategy. Read moreRead less

Often the app is developed on a single branch but the development team branches out with the addition of next features without considering how it will impact QA automation. When automating in such cases you have source control issues making it very difficult to achieve desired results.

A mobile application company focused on creating applications for healthcare providers.

The Issue: Read MoreRead less

The Company launched an app that helped physicians access updated drug and diagnostic information at point of care.

  • User adoption for the app lagged significantly as 1 in 3 users reported the application crashed after minimal use. Users reported up to 17 crashes in a day and the app's user rating in the Apple store hit rock bottom.

After a deep root analysis we determined

  • The problem was arose from how the application performed in response to user's rotating their phone which changed the screen orientation (portrait to landscape and vice versa).
  • Another issue was specifically with ios users. On ios, users accessing multiple applications at the same time was causing memory overload and leading this particular application to crash.

The Solution:Read MoreRead less

After a deep root analysis we determined that the problem was with how the application performed in response to user's rotating their phone that changed the orientation of the screen. Another issue was specifically with ios users. On ios, users accessing multiple applications at the same time was causing memory overload and leading this particular application to crash. We created automated test suite with rotation enabled testing capabilities for the client. The suite tested the app for random rotation (and if needed, could be run without rotation as well) and we worked with them closely to resolve the memory related issues.

The Impact:Read MoreRead less

After the detailed exercise, only 1 in 11 customers reported a crash and the number of crashes per user came down from 17 to 3. Their app store rating also went up to 3 stars out of 5 after the new release. Eventually, within one year of engagement with us all regression issues were caught and resolved. We automated 2000+ test case for them, enabling them run 3 different release in parallel (a huge upgrade from their earlier track record of 1 release in 3 months).
